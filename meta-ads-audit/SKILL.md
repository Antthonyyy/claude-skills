# Meta Ads — аудит и решения по креативам

Скилл отвечает на: **что крутится → какие результаты → что масштабировать, держать, наблюдать или выключить**, не сливая бюджет и **не убивая рабочие креативы по статистическому шуму**.

## Зачем родился (кейс post 9, 05.06.2026)

Я выключил **post 9 (Bali)** на основании окна «последние 7 дней»: CPL $9.26 при **5 чатах**. Применил порог отключения механически. Антон возразил — и он был прав. Понедельный разрез показал другое:

| Неделя | чаты | CPL |
|---|---|---|
| 05–11.05 | 25 | $2.53 |
| 12–18.05 | 21 | $2.70 |
| 19–25.05 | 9 | $5.77 |
| 26.05–01.06 | 10 | $5.89 |
| **Lifetime** | **69** | **$3.75** |

Это **лучший по объёму креатив на аккаунте**, плавно стареющий до всё ещё приемлемых ~$6 (при среднем по кабинету $6.57). А «$9.26» — артефакт шумной 7-дневки на 5 чатах, попавшей на худший отрезок. Плюс confound: **копия того же креатива крутилась в другой кампании (Lombok)** и растаскивала ту же аудиторию в аукционе. Реальный слабый — копия, а не оригинал. **Kill был ошибкой; откатили.**

Все правила ниже — прямое следствие этого и сопутствующих провалов сессии.

---

## Железное правило №1

**Нельзя судить креатив (особенно на килл) по одному короткому окну или малой выборке конверсий.**
Любой вердикт строится на **трёх срезах сразу**: `lifetime-агрегат` + `понедельная траектория` + `last 7d`. Снимок одного окна — не основание.

## Машина состояний

```
0 CONTEXT    — параметры кабинета: account_id, валюта, TZ, что считается конверсией, окно атрибуции
1 CONFIG     — что есть: campaigns/adsets/ads (status + effective_status + бюджет-тип + график + таргетинг + тип крео)
2 INSIGHTS   — МНОГО окон: lifetime (понедельно time_increment=7) + last 7d; пометить недозревшие дни
3 TRAJECTORY — по каждому ad: динамика CPL/CTR/CPM/freq, направление (растёт/падает/стабилизировалось)
4 CONFOUNDS  — дубли крео, пересечение аудиторий, learning, зрелость атрибуции, зомби (график/бюджет)
5 VERDICT    — keep / scale / watch / kill по фреймворку с гейтом выборки и confound'ов
6 REPORT     — отчёт: траектории, вердикты заземлены, verified vs believed, safe-now vs needs-confirm
7 EXECUTE    — пауза/килл ТОЛЬКО при прохождении гейта; reversible-first; верификация пропагации
```

## Фазы — точные действия

### 0. CONTEXT — не угадывать параметры
Из контекста проекта (`clients/<имя>/CLAUDE.md` + память) достать и **явно зафиксировать в отчёте**:
- `account_id` (act_…), **валюта** (Rawform = AUD!), **TZ кабинета** (Rawform = America/Los_Angeles).
- **Что считается конверсией**: точный `action_type` (Rawform = `onsite_conversion.messaging_conversation_started_7d` — новый чат WhatsApp), не «results» вообще.
- **Окно атрибуции** (7d/1d) — оно определяет, какие последние дни ещё дозревают.
- Целевой CPL/«коридор» по проекту.
Любая метрика в отчёте — **с валютой и окном**. «CPL 9.26» без AUD/USD и без срока — мусор.

### 1. CONFIG — снять реальное состояние
Pull по кабинету: campaigns (`status, effective_status, objective, daily_budget, lifetime_budget, start_time, stop_time, buying_type`), adsets (`targeting, optimization_goal, destination_type, promoted_object, start/end, бюджет`), ads (`effective_status, creative{video_id|object_story_spec}`).
- Зафиксировать **тип бюджета** (CBO lifetime на кампании vs daily на adset) — это меняет интерпретацию.
- Зафиксировать **тип крео** (видео/картинка/карусель) и **destination** (WhatsApp/IG DM/сайт).

### 2. INSIGHTS — многооконно (ключевая фаза)
Для **каждого ad**, который потенциально под вопросом, тянуть:
- **Lifetime** — суммарно за весь срок (агрегат CPL + всего конверсий).
- **Понедельно** — `time_increment=7` от старта до вчера: видно тренд, а не точку.
- **Last 7d** — текущий срез (но это лишь один из трёх входов, не приговор).
- **Пометить недозревшие дни**: при 7d-атрибуции последние ~3–7 дней (и **сегодня по TZ кабинета** — день не закрыт) ещё доберут конверсии. Не судить по ним как по финальным.
**Анти-cherry-pick:** окна выровнены (понедельно), не подбираются под удобный вывод.

### 3. TRAJECTORY — динамика, а не снимок
По каждому ad оценить **направление**: CPL растёт / падает / стабилизировался в коридоре. CTR (верх воронки), CPM (как аукцион оценивает релевантность), frequency (выгорание). Различать:
- **Top-funnel сломан** (CTR падает, CPM растёт) → проблема крео/релевантности.
- **Low-funnel сломан** (CTR ок, но клики не конвертят в конверсии) → оффер/посадка/насыщение сообщением.
- **Frequency высокая (≳2.0)** при росте CPL → выгорание. **Низкая freq при росте CPL** → НЕ выгорание, искать другое (сезон, confound, выборка).

### 4. CONFOUNDS — что мешает чистому выводу (обязательный чек-лист)
Перед тем как винить ad, исключить:
1. **Дубль креатива** — тот же ролик/картинка крутится в другой кампании/adset того же кабинета → **каннибализация аукциона** и расщепление конверсий. (Это и был случай post 9 ↔ Lombok-копия.)
2. **Пересечение аудиторий** — несколько adset на одну ядровую аудиторию конкурируют между собой.
3. **Learning phase** — adset/ad недавно меняли (бюджет/таргетинг/крео) → метрики нестабильны, рано судить.
4. **Зрелость атрибуции** — конверсии последних дней ещё добегают.
5. **Зомби-доставка** — `status=ACTIVE`, но **0 показов**: график закончился (`stop_time` в прошлом) и/или lifetime-бюджет исчерпан. Это не «работает плохо», это «не работает». Не путать.
6. **Малая выборка** — < минимума конверсий (см. гейт) → любой CPL статистически шумный.

### 5. VERDICT — фреймворк решения

**KEEP / SCALE** — CPL в коридоре или ниже; freq не растёт опасно; тренд стабилен/улучшается. Лидер с запасом freq → кандидат на бюджет.

**WATCH** — метрики поплыли, но **хотя бы одно** из условий килла не выполнено (мало данных, есть confound, стабилизировался в приемлемом коридоре, старение а не провал). Назначить **порог + дату переоценки**.

**KILL** — выключать ad можно **только если выполнены ВСЕ** условия:
1. **Выборка:** ≥ **8–10 конверсий** в судящем окне (не 2–5).
2. **Траектория:** CPL завышен **≥2 последовательных зрелых недель**, не одна шумная 7-дневка.
3. **Своя история:** текущий CPL **≥ ~1.8–2×** собственного стабильного базлайна ad'а **И** выше целевого/среднего по кабинету. (Дрейф $2.5→$6 при среднем $6.5 — это **старение, не провал** → WATCH, не KILL.)
4. **Confounds исключены:** нет дубля крео/пересечения аудиторий/незрелой атрибуции/learning/зомби-артефакта.
5. **Направление:** всё ещё ухудшается, не стабилизировался.
Если хоть одно не выполнено → **WATCH**, не KILL.

**Reversible-first:** предпочитать **паузу** (обратимо) удалению; на сомнении — пауза + переоценка через N дней, а не killshot.

### 6. REPORT — формат
Сохранять в `clients/<имя>/tasks/ads_audit_ДАТА.md`. Разделы:
- **Контекст** (account, валюта, TZ, конверсия, окно атрибуции) — явно.
- **Что крутится сейчас** (по `effective_status`, с пометкой зомби и реальной доставки).
- **Метрики**: агрегат + по кампаниям + по ad'ам, **с понедельной траекторией для всех спорных**.
- **Вердикты** (keep/scale/watch/kill), каждый **заземлён цифрами + прохождением гейта**.
- **Confounds**, которые учтены.
- **Рекомендации**: разделить на **safe-now** (можно делать без риска) vs **needs-confirm** (требует решения/проверки клиента — напр. качество дешёвых лидов).
- **Verified vs believed** (что проверено pull'ом vs предположение) + дисклеймер по данным (недозревшие дни, малые выборки).

### 7. EXECUTE — пауза/килл (только после гейта)
- Любая запись — после явного согласия (это правка клиентского прода).
- **READ-BEFORE-WRITE:** прочитать текущий `status`/`effective_status` перед сменой.
- После записи — **перечитать и подтвердить пропагацию**: `effective_status` досчитывается (`IN_PROCESS` → финал). `status=ACTIVE` ≠ доставка; сверить с impressions.
- Зафиксировать в отчёте, что и почему изменено, и что это обратимо.

## Hard rules (прямо из ошибок)

1. **Три окна, не одно.** lifetime + понедельно + last7d. Никогда не killдешь по одной 7-дневке.
2. **Гейт выборки.** < 8–10 конверсий = шум, вердикт только WATCH.
3. **Старение ≠ провал.** Сравнивать с собственным базлайном и средним по кабинету, не только с текущим лидером.
4. **Ищи дубль крео и пересечение аудиторий** до того, как винить ad. Каннибализация инфлейтит CPL.
5. **ACTIVE ≠ доставляет.** Проверяй effective_status + impressions; отделяй зомби (график/бюджет кончились).
6. **Атрибуция дозревает.** Не суди последние 3–7 дней и незакрытый день TZ как финал.
7. **Валюта + окно всегда.** Метрика без них — мусор. Для клиента в USD — конвертировать явно.
8. **Reversible-first.** Пауза, не удаление; на сомнении — отложить и переоценить.
9. **Заземление по реальному pull'у.** Вердикт «в воздухе» (без чисел из кабинета) не публиковать.
10. **Рекомендации делить** на safe-now vs needs-confirm; не выдавать гипотезу за факт.

## Pull-паттерн (Graph API v21, токен из `.env`)

Существующий `scripts/meta_executor.py pull` и `campaign_details.py` тянут **одно окно** — для этого скилла **недостаточно**. Минимальный многооконный сниппет:

```python
import json, urllib.request, urllib.parse
tok=next(l.split("=",1)[1].strip().strip('"') for l in open(".env") if l.startswith("META_ACCESS_TOKEN="))
ACC="act_XXXXXXXXXX"; CONV="messaging_conversation_started"  # подставить под проект
def g(p,**f):
    f["access_token"]=tok
    with urllib.request.urlopen(f"https://graph.facebook.com/v21.0/{p}?"+urllib.parse.urlencode(f),timeout=60) as r: return json.load(r)
def chats(acts):  # вытащить нужную конверсию
    return next((int(float(a["value"])) for a in acts or [] if CONV in a.get("action_type","")),0)

# понедельная траектория одного ad (ключ к анти-shum)
ins=g(f"{AD_ID}/insights", time_range=json.dumps({"since":START,"until":YESTERDAY}),
      time_increment="7", fields="spend,impressions,frequency,ctr,cpm,actions", limit="20")
# + отдельным вызовом lifetime (без time_increment) и last7d
```
Ключевое: `time_increment="7"` даёт **понедельный разрез** (а не снимок), `effective_status` в `ads`-запросе ловит зомби, `actions` фильтруется по точному `action_type` конверсии.

## Антипаттерны (что НЕ делать)

- ❌ Тянуть один `last_7d` и выносить вердикт по нему.
- ❌ Killить ad по 2–5 конверсиям.
- ❌ Применять чужой порог отключения механически, без своей истории ad'а и confound-чека.
- ❌ Считать `status=ACTIVE` за «работает».
- ❌ Сравнивать ad только с текущим лидером, игнорируя его собственный трек.
- ❌ Report без валюты/TZ/окна атрибуции.
- ❌ Выключать без согласия и без верификации пропагации.
