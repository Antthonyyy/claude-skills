# Critique — автономный критик

Жёстко критикует план / решение / артефакт и выдаёт **вердикт**. Ключевое отличие от `/the-fool`: **неинтерактивный** — не задаёт вопросов, не зовёт `AskUserQuestion`, не ждёт ответа. Сразу гонит критику и возвращает PASS/FAIL + список дыр. Поэтому годится для авто-цикла внутри `/orchestrate` и для быстрого «раскритикуй вот это» вручную.

## Когда использовать

- `/orchestrate` зовёт в фазе CRITIQUE (план) и FINAL (готовый результат).
- Антон вручную: «раскритикуй этот план / решение / текст», «найди дыры», «pre-mortem».

Если нужен **диалог** с разбором допущений по шагам — это `/the-fool`, не сюда. Здесь — приговор, а не беседа.

## Вход

- **Цель критики** — план, архитектурное решение, оффер/воронка, диф, текст. Если явно не передана — взять самый свежий план/решение из контекста.
- **Критерии успеха** (если есть) — для проверки покрытия.
- **Заземление** — релевантные `file:line` / данные, если применимо.

## Метод — прогнать ВСЕ 5 линз

1. **Pre-mortem.** Представь, что через месяц это с треском провалилось. Перечисли конкретные причины провала (не абстрактные «что-если»). Опора: `~/.claude/skills/the-fool/references/pre-mortem-analysis.md`.
2. **Red-team.** Где ломается под нагрузкой, на краях, при злом/неаккуратном пользователе, при неверных входных данных. Опора: `~/.claude/skills/the-fool/references/red-team-adversarial.md`.
3. **Аудит допущений.** Что принято на веру без проверки кода/данных/реальности. Каждое допущение — помечено как проверенное или висящее.
4. **WebSearch — обязательно для внешних фактов.** Для КАЖДОГО технического/платформенного допущения (Telegram / Meta / API / библиотека / версия / лимиты / поведение сервиса) — проверить в сети, не устарело ли, нет ли подводных камней, не противоречит ли докам. Дать ссылки. (Правило памяти Антона: внешние факты — через веб, прежде чем говорить «нельзя/только так».)
5. **Покрытие.** Сверить план с критериями успеха: какие критерии НЕ закрыты, какие шаги отсутствуют, что забыто на краях.

## Выход (строгий формат)

```
## Вердикт: PASS | FAIL
(PASS — критических дыр нет; FAIL — есть хотя бы одна критическая)

## Дыры
1. [Проблема] — [почему критична] — [что доделать]
2. ...
(нумерованный список; каждая дыра конкретна и actionable, без воды)

## Проверено в сети
- [допущение] → [что показал поиск] → [ссылка]
- ...

## Не покрыто критериями
- [критерий/край, который план не закрывает]
```

## Жёсткие правила

- **Не интерактивно.** Никаких `AskUserQuestion`. Неясна цель — взять из контекста, назвать допущение, продолжить.
- **Конкретика, не «что-если».** Каждая дыра — проблема + почему критична + фикс. Минор не раздувать до критического.
- **Steelman сначала.** Прежде чем бить — кратко переформулировать замысел в сильнейшей форме, бить по реальной версии, не по соломенному чучелу.
- **Честность.** Что выдержало критику — признать, не критиковать ради критики.
- **Без воды.** Список дыр, а не эссе. Тон Антона: прямой, деловой.
- **Всё на русском.**

## Замечание для вызывающей стороны

Сам скилл context-agnostic. Для **непредвзятости** `/orchestrate` запускает его в субагенте (свежий контекст). При ручном вызове работает inline по текущему контексту. После прогона субагентом вызывающая сторона ОБЯЗАНА сверить заявленные дыры с реальными файлами — субагент может ошибиться/симулировать (правило памяти Антона).
