# Orchestrate — дирижёр задач

Один скилл-дирижёр. Берёт задачу Антона и ведёт её через машину состояний:
**план → автономная критика (red-team + pre-mortem + WebSearch) → переписать → исполнение по шагам с gate после каждого → финальная сверка.** Не переписывает существующие движки — вызывает их по фазам.

## Когда использовать

- Антон даёт нетривиальную задачу (фича, рефактор, баг, стратегия, воронка, медиаплан) и хочет, чтобы её прогнали «нормально по этапам», а не сделали с наскока.
- Любая задача, где цена ошибки выше, чем стоимость лишнего круга критики.

**НЕ использовать** для: однострочных правок, тривиальных вопросов, чистого чтения/поиска. Там оверхед цикла не оправдан — делать напрямую.

## Машина состояний

```
ЗАДАЧА
  │
  ▼ 0 CAPTURE   — зафиксировать задачу + verifiable-критерии; заземлить по дереву (file:line)
  ▼ 1 PLAN      — N шагов, каждый = действие + проверка → TodoWrite
  ▼ 2 CRITIQUE  — субагент, fresh ctx: red-team+pre-mortem+WebSearch → PASS/FAIL + список дыр
  │   FAIL → 3 REVISE (переписать под дыры) → назад в 2   (макс 3 круга)
  │   PASS → ЧЕКПОЙНТ: показать финальный план Антону, дождаться ОК
  ▼ 4 EXECUTE   — по каждому шагу: (a) сделать в режиме Karpathy
  │                                (b) GATE «идём ли верно»
  │                                    PASS → следующий шаг
  │                                    FAIL → починить ровно дыры → повтор (макс 2 ретрая → эскалация)
  ▼ 5 FINAL     — субагент, fresh ctx: ничего ли не потеряно, все критерии (0) закрыты
  ▼ ОТЧЁТ → clients/имя/tasks/ (или personal/имя/tasks/)
```

## Фазы — точные действия

### 0. CAPTURE
- Восстановить из задачи: что именно нужно, в какой папке проекта (`clients/` или `personal/`, см. `ROUTING.md`), какой это домен — **код** или **маркетинг** (определяет gate в фазе 4).
- Сформулировать **verifiable-критерии успеха** (что именно проверим в конце). Без них фаза 5 невозможна.
- **Заземлить по реальному дереву**: прочитать релевантные файлы, зафиксировать `file:line`. План «в воздухе» не публиковать (правило Антона).
- **Развилки-решения пользователя задавать СРАЗУ (фаза 0), не копить до чекпойнта.** Если задача требует выбора, который знает только Антон (источник данных, доступ/токен, объём, числовой id vs username) — спросить здесь, через `AskUserQuestion`. Откладывание на чекпойнт = лишний заход.

### 1. PLAN
- Разбить на N шагов. **Каждый шаг = действие + конкретная проверка**: `1. [действие] → проверка: [что покажет, что шаг закрыт]`.
- Шаги — минимальный достаточный набор (Karpathy: ничего спекулятивного).
- Занести план в `TodoWrite` — это исполнительный костяк фазы 4.

### 2. CRITIQUE — через `/critique` в субагенте (fresh context)
Запустить **субагентом** (`Agent`, subagent_type `general-purpose`) с задачей: «Применить скилл `/critique` к этому плану + критериям + ключевым file:line». Свежий контекст = непредвзятый взгляд. `/critique` неинтерактивен — сам прогоняет pre-mortem + red-team + аудит допущений + WebSearch + покрытие и возвращает вердикт **PASS/FAIL** + нумерованный список дыр (формат — в `~/.claude/skills/critique/SKILL.md`).

**После субагента — проверить его артефакт сам** (правило памяти: субагенты могут симулировать вызовы). Сверить заявленные дыры с реальными файлами, отбросить ложные.

### 3. REVISE
- Переписать план, закрыв **каждую** дыру из вердикта FAIL. Обновить `TodoWrite`.
- Вернуться в фазу 2. **Лимит — 3 круга.** Если после 3-го всё ещё FAIL — остановиться и эскалировать Антону с остатком дыр (не зацикливаться).

### ЧЕКПОЙНТ (после PASS)
Показать Антону финальный план + что критика нашла и как закрыто. Дождаться «ОК». Дальше — автономно до конца.

### 4. EXECUTE — по шагам, режим Karpathy
Для каждого todo по очереди:
- **(a) Сделать шаг в режиме Karpathy guidelines:** думать до кода, хирургические изменения (трогать только нужное, не «улучшать» соседнее), минимум кода, убрать сирот от своих правок, совпадать со стилем. Это обязательный режим письма (CLAUDE.md: дефолт для всех задач с кодом).
  - **READ-BEFORE-OVERWRITE для прод-состояния.** Перед перезаписью любого существующего значения на проде (env-переменная, конфиг, запись в БД) — СНАЧАЛА прочитать текущее значение. Для list/CSV-переменных (например `WNW_MANAGER_IDS`) дописывать к существующему merged-набору, не затирать. `.env` ≠ прод-env — источник истины это прод. Урок 2026-06-02: `railway variables --set` стёр уже стоявшего менеджера.
- **(b) GATE «идём ли верно»** — по домену:
  - **код:** `/code-reviewer` по дифу шага + при наличии рантайма `/verify`; сверить с проверкой из плана.
  - **маркетинг:** сверить результат шага с критерием шага; при риске — короткий автономный the-fool по артефакту.
  - PASS → пометить todo done, следующий шаг.
  - FAIL → починить **ровно** названные дыры (не расширять scope) → повтор GATE. **Лимит 2 ретрая**, дальше эскалация Антону с тем, что не закрылось.

### 5. FINAL REVIEW — fresh context
Запустить субагентом по полному дифу/набору артефактов:
- Все ли verifiable-критерии из фазы 0 закрыты? Перечислить по пунктам.
- Не потеряли ли что-то из шагов плана (регресс, забытый край)?
- `/code-reviewer` по полному дифу; `/security-review` — если трогали auth / БД / external API.
- `/critique` (в субагенте) по готовому результату: «что всё ещё хрупко, какие критерии не закрыты».
- **Отчёт делить на «verified» (проверено запуском/чтением) vs «believed done» (сделано, но не подтверждено)** — правило памяти Антона.

### ОТЧЁТ
Сохранить в `clients/имя/tasks/orchestrate_<задача>_ДАТА.md` (или `personal/`): задача, критерии, что нашла критика + как закрыто, что сделано по шагам, финальная сверка (verified/believed), остаточные риски.

## Жёсткие правила (из памяти Антона — нарушать нельзя)

- **Заземление по коду** (`file:line`) до публикации плана. План в воздухе — не план.
- **Проверка артефактов субагентов** — после делегата `ls`/Read сам; критику/финалу на слово не верить.
- **verified vs believed-done** — в каждом отчёте раздельно.
- **Лимиты петель** — критика ≤3 круга, ретрай шага ≤2. Дальше — эскалация Антону, не зацикливаться.
- **Карта задач** — `ROUTING.md`: личное (`personal/`) ↔ работа (`clients/`). Артефакты в `tasks/`.
- **Внешние факты — через WebSearch** прежде чем уверенно сказать «нельзя/только так».
- **Всё на русском**, включая код и комментарии.

## Что НЕ делает

- Не лезет в `wnw crm` (`wnw-api`, `wnw-cron`, frontend) — отдельный репо.
- Не деплоит сам без явной команды; Railway — только после гейта аккаунта (клиентский `a81373a6`).
- Не расширяет scope шага при починке gate-фейла.
