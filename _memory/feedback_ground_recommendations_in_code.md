Любой план/рекомендация по выполнению задачи **должна быть заземлена по реальному коду**, прежде чем считаться готовой. Антон лично проверяет планы по дереву и режет рекомендации, которые «висят в воздухе».

**Why:** 25.05.2026 я отдал план улучшения ИИ-продавца WNW — после прогона через analyst+strategist агентов. Стратег НЕ смог прочитать knowledge/bot/*.md (флака среды) и его рекомендации были по предполагаемой структуре. Антон сам проверил по коду и нашёл 6 фактических ошибок: захардкоженная таймзона была неверной для лета, монолит-fallback пропущен из плана, sanitize_first_name на самом деле возвращает мусор, salutation требует смены контракта а не чистки имени, P1 нужно править тезис а не только списки триггеров. Все — отлавливались чтением кода до публикации плана.

**How to apply:**
1. Перед финализацией плана/рекомендации — прочитать реальные функции/модули/строки, на которые ссылаешься. Не доверять описанию от субагента и не делать предположений о структуре. Особенно проверять: точные имена функций (например `is_telegram_auto_reply` в tg_outreach.py:617, не в outreach_core), точное содержимое fallback-веток, дефолтные значения (`sanitize_first_name(fallback="там")` → ломает шаблоны), таймзонозависимые значения (вычислять через zoneinfo, не хардкодить).
2. Указывать `file:line` ссылки везде, где упоминаешь код — Антон проверит, лучше упредить.
3. Различать «контракт» (`{name_prefix}` placeholder) и «чистку» (`sanitize_first_name`). Если корень в контракте — патч чистки не решит. Связано с [[feedback_subagents_may_simulate_tool_calls]].
4. Для двухслойных систем (код + промт + fallback) — план должен явно адресовать все слои или явно объявить часть out-of-scope. См. [[project_wnw_bot_prompt_architecture]].
