Инициатива (старт 2026-05-28): авто-ingestion новых членов клуба WNW из TG-чата **«Новенькие»** (id 3441667525) в CRM, чтобы снять с менеджеров ручное заполнение.

**Why:** в чат постят новых резидентов/экспертов клуба + оплаты пакетов; сейчас Татьяна/менеджеры вносят это в CRM руками. Цель — один тап подтверждения вместо ручного создания contact+membership+payment.

**PIVOT 2026-05-28 (по реальному экспорту чата):** это НЕ event-attendee, а **членство/подписка**. Поток ДВУХШАГОВЫЙ: (1) пост-кандидат [284,449,524…] даёт пакет/цену/контакты, но НИКОГДА не делает paid=true; (2) **реплай админа** (Татьяна 652907258 / WNW 1865067456) подтверждает оплату. «не вижу»/«от кого эта оплата?» → not_confirmed. Пакеты: эксперт 170/260/290€, спикер 450/520€, резидент города 150€, 6 мес.

**How to apply:** при возврате к этой задаче — НЕ переизобретать архитектуру, она залочена с Антоном.

- CRM владеет `status/reviewed_by/decision` + apply in-process. → 1 источник истины, 0 reconcile, 0 split-brain.

### Залоченные решения
- **Reader = groupmon userbot** (`scripts/group_monitor.py`, `TG_GROUPMON_STRING_SESSION`), НЕ @Gnilosirbot и НЕ outreach-аккаунт (см. [[MEMORY]] изоляция аккаунтов). Bot API только для карточек ревью.
- **Write-path = через wnw-api** (CRM — единственный владелец платежей), новый эндпоинт `POST /intake/club-member` в репо **Antthonyyy/wnw-api** (отдельный git, чекаут в `wnw crm/wnw-api/`, синхрон с origin/main на c910d56), под `_api`/X-Api-Secret. Разрешает прошлые даты, reject cancelled. Существующих эндпоинтов не хватает: `register` требует `ss_id` + только upcoming; `payment/manual` нужен готовый `registration_id`.
- **Цель = membership/subscription:** upsert contact по phone/TG/email (НЕ по имени) → обновить `contacts.package_type/is_resident/package_*` → при ПОДТВЕРЖДЁННОЙ оплате insert `payments.purpose='subscription'`. НЕ event_registrations.
- **package_type в v1 НЕ расширяем** (в CRM фактически TEXT, но UI/типы/роуты ждут `expert|expert_speaker`): эксперт/резидент города/спец-тарифы 170/260/290/150 → `expert`; спикер 450/520 → `expert_speaker`. Сырьё хранить отдельно: package_raw, package_price_cents, paid_amount_cents, expected_amount_cents, duration_months, city (в intake/metadata/notes).
- **Нетворкинг-ростеры — ОТДЕЛЬНЫЙ seed-трек** (`contact_seed_intake`: name+tg+city, без денег/пакета/resident-мутаций). В payment-пайплайн не класть.
- **v1 — только ручное подтверждение** каждого платежа (ноль авто-применения денег). Порог авто — только в v2 после замера точности.
- ~~**Ревью — карточки в @Gnilosirbot** с кнопками~~ → **SUPERSEDED** (см. АРХ-РЕШЕНИЕ #2): ревью в CRM-дашборде, apply in-process.
- 4 отдельных компонента: A reader-форк в group_monitor (до intel/digest, чтобы PII не утёк) · B `payment_intake` (новая таблица+модуль, UPSERT DO UPDATE, статусы detected→needs_review→approved→applied) · C callback-service · D wnw-api endpoint.

### Hard-констрейнты (из pre-mortem, без них провал)
- Телефон → E.164 через `phonenumbers` (нет в requirements.groupmon.txt — добавить), region-default по городу. Фундамент дедупа.
- Имя без телефона / скрин / нет точного матча → принудительное ручное ревью, никогда не авто.
- **paid только по подтверждению whitelisted-админа** (Татьяна 652907258 / WNW 1865067456), желательно reply-связкой. Пост кандидата → максимум `payment_terms {full,partial,invoice,unknown}`, не paid. Partial → пишем фактическую сумму (paid_amount), не притворяясь полной.
- Идемпотентность: intake.unique(source_chat_id,source_msg_id) → payments unique partial по `source_reference="payment_intake:<intake_id>"` (нужна миграция в wnw-api — сейчас индекс неуникальный).
- Детект только вперёд от `payment_cutoff_msg_id`/`payment_cutoff_at` в monitored_chats.yaml (catch-up не тянет историю).
- Контакт в wnw-api создаётся новым helper'ом через `_find_contact_cross_channel` (НЕ `_upsert_contact` — он ss_id/smartsender-coupled), source="payment_intake".
