При любом коммите/мерже в этом репо от лица Антона:

- **Никогда `git add -A` / `git add .`** — стейджить только явными путями.
- **Показать staged-набор и проверить** его до коммита (что не попало чужое/WIP/секреты).
- **Несвязанные изменения — на отдельную ветку** (напр. реорг репо → `chore/...`, не в фича-ветку).
- **Перед merge проверять расхождение веток** (`main..feat`), не мержить вслепую — иначе можно затащить лишние коммиты или словить конфликты.
- **Точность в формулировках git:** «moved» ≠ «deleted» (перемещение tracked-файла без `git add` нового пути = удаление из истории); «синхронизировал» только если реально 1:1.

**Why:** у Антона почти всегда есть активная фича-ветка с незакоммиченным WIP вперемешку (feat/tg-group-monitor была на 26 коммитов впереди main с кучей uncommitted-правок). Слепой `git add -A` смешал бы реорг с фичей, мог затащить PII-CSV или секреты. Он сам подсветил риск `git add -A` и поправил неточные формулировки в моём отчёте. Также `.env`/`.env.*`/`credentials/**` тут **зашифрованы git-crypt и коммитятся** (не gitignore) — не пытаться их untrack'ать.

**How to apply:** когда коммичу/мержу — стейдж по явным путям → показать `git diff --cached --name-status` → подтвердить ветку → для merge проверить FF/divergence. Связано с [[feedback-verification-discipline]] и [[feedback-ground-recommendations-in-code]].
